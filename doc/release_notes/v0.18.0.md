# Release Notes v0.18.0

## Highlights

This minor release introduces `ds_connector_create.sh`, a new script that
handles the complete lifecycle of creating an OCI Data Safe On-Premises
Connector — from the OCI API call all the way through local installation and
optional systemd service registration. Two utility functions
(`is_valid_bundle_key()` and `generate_bundle_key()`) have been extracted from
`ds_connector_update.sh` into the shared library so both update and create
scripts share the same implementation.

- **New**: `bin/ds_connector_create.sh` — full end-to-end connector creation.
- **Refactored**: bundle-key helpers moved to `lib/oci_helpers.sh`.
- **New OCI helper**: `ds_create_connector()` added to `lib/oci_helpers.sh`.

---

## Added

### `bin/ds_connector_create.sh` — end-to-end On-Premises Connector creation

The new script automates the full connector provisioning workflow in six steps:

1. **Create** the OCI Data Safe connector object
   (`oci data-safe on-prem-connector create`)
2. **Bundle key** — reuse an existing key from `etc/<name>_pwd.b64`, or
   generate a fresh one with `--force-new-bundle-key`
3. **Download** the installation bundle
   (`oci data-safe on-prem-connector generate-on-prem-connector-configuration`)
4. **Create** the local connector home directory
5. **Extract** the installation bundle
6. **Run** `setup.py install` with the bundle key to activate the connector

Optional post-install steps (both are skipped by default):

- `--wait-state STATE` — poll until the connector reaches `STATE` (default:
  `ACTIVE`; pass `--wait-state ""` to return immediately)
- `--register-oradba ENV` — register the connector in `oradba_homes.conf`
  under the given OraDBA environment name
- `--install-service` — delegate to `install_datasafe_service.sh` to install
  a systemd service for the connector (requires root)

#### Required arguments

| Flag | Description |
| ---- | ----------- |
| `-c, --compartment ID` | Compartment OCID or name |
| `-N, --display-name NAME` | Connector display name in OCI Data Safe |
| `--connector-home PATH` | Local installation directory (must not exist yet) |

#### Optional arguments

| Flag | Description |
| ---- | ----------- |
| `--description DESC` | Free-text description for the OCI connector object |
| `--force-new-bundle-key` | Always generate a fresh bundle key (never reuse) |
| `--wait-state STATE` | Poll until STATE is reached (default: `ACTIVE`) |
| `--register-oradba ENV` | Register in `oradba_homes.conf` as ENV |
| `--install-service` | Install systemd service after setup |
| `-n, --dry-run` | Show plan without making changes |

#### Config

| Variable | Description |
| -------- | ----------- |
| `DS_ROOT_COMP` | Default compartment (OCID or name) |
| `DS_CONNECTOR_COMP` | Alternative compartment for connectors |

#### Examples

```bash
# Dry-run: show what would be created
ds_connector_create.sh -c my-compartment -N my-connector \
    --connector-home /u01/datasafe/my-connector --dry-run

# Create and wait until ACTIVE
ds_connector_create.sh -c my-compartment -N my-connector \
    --connector-home /u01/datasafe/my-connector

# Create without waiting (async)
ds_connector_create.sh -c my-compartment -N my-connector \
    --connector-home /u01/datasafe/my-connector --wait-state ""

# Create with all post-install steps
ds_connector_create.sh -c my-compartment -N my-connector \
    --connector-home /u01/datasafe/my-connector \
    --register-oradba dscon5 --install-service

# Create with explicit description and fresh bundle key
ds_connector_create.sh -c my-compartment -N my-connector \
    --connector-home /u01/datasafe/my-connector \
    --description "Production connector for Exadata X9M" \
    --force-new-bundle-key
```

### `lib/oci_helpers.sh` — shared bundle-key helpers and new OCI wrapper

#### `is_valid_bundle_key(key)`

Validates that a bundle key meets OCI's complexity requirements:

- 12–30 characters
- At least one uppercase letter
- At least one lowercase letter
- At least one digit
- At least one special character (`!@#$%^&*()_+`)

Returns 0 if valid, 1 otherwise.

#### `generate_bundle_key()`

Generates a random 20-character bundle key that satisfies OCI complexity rules,
using `openssl rand -base64 64` as entropy source. The key is printed to stdout.

#### `ds_create_connector(comp_ocid, display_name [, description])`

Wraps `oci data-safe on-prem-connector create`, waits for the asynchronous work
request to settle, and returns the new connector OCID on stdout. Calls `die()`
on failure.

## Changed

### `bin/ds_connector_update.sh` — removed duplicate helpers

`is_valid_bundle_key()` and `generate_bundle_key()` were previously defined
inline in `ds_connector_update.sh`. They have been removed from that script and
are now sourced from `lib/oci_helpers.sh` via `ds_lib.sh`. No behaviour change.

## Fixed

### `scripts/build.sh` — stale release note on upgrade

The build script previously staged the current release note as
`doc/v${VERSION}.md` inside the tarball. After an upgrade (e.g. v0.17.6 →
v0.18.0), the old `doc/v0.17.6.md` remained on disk alongside the newly
installed `doc/v0.18.0.md`. The `.extension.checksum` for the new version only
lists `doc/v0.18.0.md`, so the unexpected leftover file caused checksum
verification to fail.

**Fixed**: the release note is now staged as `doc/RELEASE_NOTES.md` (a fixed
name). The file is overwritten on each upgrade so only one copy ever exists.

---

## Upgrade Notes

No configuration changes are required. The new script is additive; existing
workflows are unaffected.

**Note**: if you are upgrading from v0.17.5 or v0.17.6, you may have a leftover
`doc/v0.17.x.md` file in your installation directory. It can be safely removed:

```bash
rm -f "${ORADBA_BASE:-/opt/oracle/local/odb_datasafe}/doc/v0.17.*.md"
```
