# Release Notes - v1.0.6

**Release Date:** 2026-07-01
**Type:** Patch (bug fix - ds_target_reregister camelCase/kebab-case key conflict)

## Summary

v1.0.6 fixes a data corruption bug in `ds_target_reregister.sh` where the OCI
`--database-details` JSON passed to `target-database update` contained duplicate
conflicting keys, causing OCI CLI to reject the request with exit 1.

## Changes

### Fixed: `compute_new_db_details()` - Duplicate Key Conflict

`compute_new_db_details()` builds a patch object and merges it into the current
`database-details` fetched from OCI via `jq '. + $patch'`. OCI CLI returns
`database-details` fields in kebab-case (`vm-cluster-id`, `service-name`,
`listener-port`). The patch was built using camelCase keys (`vmClusterId`,
`serviceName`, `listenerPort`).

Since `jq '. + $patch'` only overwrites keys with identical names, the result
contained both the old kebab-case values (from the current OCI state) and the
new camelCase values (from the patch) simultaneously:

```json
"vm-cluster-id": "ocid1.vmcluster...OLD...",
"vmClusterId":   "ocid1.vmcluster...NEW...",
"service-name":  "old-service.example.com",
"serviceName":   "new-service.example.com"
```

OCI CLI rejected the ambiguous JSON (exit 1) with "Structural update failed".

**Fix:** Patch keys changed to kebab-case throughout `compute_new_db_details()`
so they match the OCI response format and `jq '. + $patch'` correctly overwrites
the existing values.

**Side effect fixed:** `show_reregister_plan()` read `."vm-cluster-id" //
.vmClusterId`, which resolved to the OLD cluster OCID (kebab-case evaluated
first). The dry-run output therefore appeared unchanged for cluster/service,
masking the duplicate-key bug. After the fix the plan correctly shows the new
values.

## Upgrade Notes

No API or configuration changes. Update `ds_target_reregister.sh` in place.
Any pending `--cluster` + `--from-oci` re-registration that previously failed
with exit 1 can be retried after updating.
