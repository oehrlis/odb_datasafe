# Release Notes - v1.0.1

**Release Date:** 2026-06-29
**Type:** Patch (installer hardening - batch mode, service-stop robustness, thin-wrapper refactor)

## Summary

v1.0.1 delivers targeted hardening of the service installer pair. The headline
change is `--all` batch mode in `install_datasafe_service.sh`, which lets a single
root invocation prepare, install, uninstall, or check every discovered connector.
Alongside this, a new `stop_service()` function centralises the stop logic (prefer
`oradba_dsctl.sh stop`, fall back to `systemctl stop` + pkill of CMAN processes),
and `uninstall_all_datasafe_services.sh` is refactored from a 548-line standalone
to a ~230-line thin wrapper that delegates per-connector uninstall to
`install_datasafe_service.sh --uninstall`.

A companion change tightens the install path: `install_service()` no longer
silently regenerates the service file when the OS user differs; it now raises a
hard error if the prepared service file is missing, forcing an explicit `--prepare`
run first.

## Changes

### New: `--all` Batch Mode

`install_datasafe_service.sh` now accepts `--all` to process every connector
discovered under the connector base directory in a single invocation:

```bash
# Prepare all connectors (as oracle user)
./install_datasafe_service.sh --prepare --all

# Install all connectors (as root)
sudo ./install_datasafe_service.sh --install --all

# Check all connectors
sudo ./install_datasafe_service.sh --check --all

# Uninstall all connectors (as root)
sudo ./install_datasafe_service.sh --uninstall --all
```

`--all` is mutually exclusive with `-n`/`--connector` (combining them is a hard
error). It implies non-interactive mode and prints a batch summary on completion.

### New: `resolve_install_context()` Function

For `--install`, `--uninstall`, and `--check` operations, the installer now calls
`resolve_install_context()` instead of the full `validate_connector()`. This skips
`JAVA_HOME` and CMAN directory validation (those paths are embedded in the already-
prepared service file) and instead reads `OS_USER` and `OS_GROUP` directly from
the prepared service file. If `--user` is supplied on the CLI and differs from the
`User=` field in the prepared file, a warning is emitted rather than silently
overriding.

**Effect:** `sudo install_datasafe_service.sh --install -n exacc-wob-vwg-ha1` no
longer fails with `JAVA_HOME not found` when root's environment lacks `ORACLE_BASE`
or `JAVA_HOME`.

### New: `stop_service()` Function

A dedicated `stop_service()` function centralises connector shutdown:

1. If `oradba_dsctl.sh` is available, use `oradba_dsctl.sh stop <alias>`.
2. Otherwise, call `systemctl stop` and follow with `pkill` of residual CMAN
   (`cmgw`) processes.

`uninstall_service()` now calls `stop_service()` instead of bare `systemctl stop`.

### Changed: `install_service()` - Hard Error on Missing Prepared File

The auto-regenerate block (automatic `prepare_service` call when the service file's
`User=` did not match `--user`) has been removed. `install_service()` now exits
with an error when the prepared service file is absent, requiring an explicit
`--prepare` phase first. This enforces the documented two-phase workflow and
prevents root from needing connector-environment knowledge (`JAVA_HOME`, `ORACLE_BASE`).

### Refactored: `uninstall_all_datasafe_services.sh`

`uninstall_all_datasafe_services.sh` shrinks from 548 lines to ~230 lines. The
refactored script:

- Still discovers installed services via `systemctl list-unit-files` (catches
  orphaned services with no corresponding connector directory).
- Delegates per-connector work to `install_datasafe_service.sh --uninstall -n <conn>`.
- No longer duplicates `stop_service()` logic.

### Test Changes

- **REG-003** updated: now validates that `--install` emits a `WARNING` when
  `--user` differs from the `User=` field in the prepared service file (auto-
  regeneration behaviour removed); also verifies the prepared file is not modified.

## Upgrade Notes

- **Prepared service file is now mandatory for `--install`.** If you previously
  relied on the auto-regenerate behaviour (calling `--install` without a prior
  `--prepare`), add an explicit `--prepare` step first.
- **OS user is read from the prepared file.** Passing `--user` to `--install`
  when a prepared file already exists will warn, but the file's `User=` takes
  precedence. Re-run `--prepare` with the desired `--user` if you need to change it.
- **`uninstall_all_datasafe_services.sh` requires `install_datasafe_service.sh`.**
  Both scripts must be deployed together; the wrapper delegates to the installer.
