# Release Notes v0.18.2

## Highlights

This patch release fixes three issues: two with `ds_connector_create.sh`
registration, a major correctness bug in `ds_target_connector_summary.sh`, and
`--targets` being silently ignored in `ds_target_update_connector.sh distribute`.

- **Fixed**: `ds_connector_create.sh --register-oradba` failed with
  `Unknown option: --connector-home` and could not register connectors whose
  entry did not yet exist in `oradba_homes.conf`.
- **Fixed**: `ds_target_connector_summary.sh` showed on-prem connector targets
  in "Cloud / Private Endpoint" because the OCI `target-database list` API
  omits `connection-option` for some targets. Post-grouping enrichment now
  fetches per-target details only for the affected subset.
- **Fixed**: `ds_target_update_connector.sh distribute --targets LIST` silently
  ignored the target list and distributed the entire compartment instead.
- **Added**: `--no-enrich` flag to skip enrichment (fast output, useful for
  troubleshooting to identify which targets are affected).
- **Added**: connector groups are now sorted alphabetically by connector name
  in all output formats.

---

## Fixed

### `--register-oradba` fails with `Unknown option: --connector-home`

`ds_connector_create.sh` passed `--connector-home PATH` to
`ds_connector_register_oradba.sh`, which does not accept that option, causing:

```text
[ERROR] Unknown option: --connector-home (use --help for usage)
[WARN]  OraDBA registration failed (non-fatal)
```

**Fixed**: removed the unsupported `--connector-home` argument from the call.

### `--register-oradba` fails when entry is absent from `oradba_homes.conf`

If the named environment (e.g. `dscon6`) did not already have an entry in
`oradba_homes.conf`, `ds_connector_register_oradba.sh` exited with:

```text
[ERROR] DataSafe environment 'dscon6' not found in oradba_homes.conf
```

**Fixed**: `optional_register_oradba()` now checks whether the entry exists
before calling the registration script. If it is missing, it invokes
`oradba_homes.sh add` to create it:

```bash
oradba_homes.sh add \
    --name dscon6 \
    --path /u01/datasafe/ds-conn-exacc-wob-vwg-ha6 \
    --type datasafe \
    --desc "ds-conn-exacc-wob-vwg-ha6"
```

The connector display name becomes the description; `ds_connector_register_oradba.sh`
then appends `(oci=<OCID>)` to it, resulting in:

```text
dscon6:/u01/datasafe/ds-conn-exacc-wob-vwg-ha6:datasafe:50:dscon6:ds-conn-exacc-wob-vwg-ha6 (oci=ocid1...):AUTO
```

`oradba_homes.sh` is located via `command -v oradba_homes.sh` (PATH) with
`${ORADBA_BASE}/bin/oradba_homes.sh` as fallback.

### On-premises connector targets misclassified as "Cloud / Private Endpoint"

The OCI `target-database list` API does not return `connection-option` for
some targets, causing all of them to fall through to the "no-connector" group
during the initial jq grouping pass — even though each target has a real
on-premises connector when queried individually.

Previously `enrich_targets_with_connector()` made one `target-database get`
call per target (potentially 1 475 calls). That function was removed in an
earlier refactor, leaving the 135 affected targets unresolved.

**Fixed**: a targeted post-grouping enrichment step now:

1. Identifies non-DELETED targets in the "no-connector" group after initial
   grouping (typically ~135 in a large deployment).
2. Fetches `connection-option` and `associated-resource-ids` for each via
   individual `target-database get` calls.
3. Merges the enriched fields back into the in-memory target data.
4. Looks up names for any newly visible connector OCIDs not already in the
   connector map.
5. Re-groups all targets with the enriched data.

DELETED targets remain in "Cloud / Private Endpoint" as expected (they have
no active connector).

### `distribute --targets` silently ignored the target list

`ds_target_update_connector.sh distribute` parsed `-T/--targets` but
`do_distribute_mode()` always called `list_targets_in_compartment "$COMPARTMENT"`,
completely discarding the provided target list. All targets in the compartment
were distributed instead of only the specified ones.

**Fixed**: `do_distribute_mode()` now mirrors the pattern used by `do_set_mode()`:
when `$TARGETS` is set, it calls `ds_collect_targets "$COMPARTMENT" "$TARGETS"
"$LIFECYCLE_STATE" "$TARGET_FILTER"` instead of listing the entire compartment.

```bash
# Distribute only specific targets — now works correctly
ds_target_update_connector.sh distribute \
    --targets target-db-01,target-db-02,target-db-03 --apply

# Distribute with a regex filter
ds_target_update_connector.sh distribute -r "^prod-" --apply
```

### Spurious `[WARN] Ignoring unexpected arguments:` with empty content

`parse_common_opts` could leave whitespace-only strings in the positional
argument list. The previous guard `[[ -n "$arg" ]]` did not reject them,
producing a spurious warning. Fixed by stripping whitespace before the check:
`[[ -n "${arg//[[:space:]]/}" ]]`.

## Added

### `--no-enrich` flag for `ds_target_connector_summary.sh`

Skip the post-grouping enrichment pass entirely. Targets whose connector info
is absent from the list response will appear as "Cloud / Private Endpoint".
Useful for troubleshooting (identify affected targets quickly) or when fast
output is more important than accuracy:

```bash
# Fast — see which targets are affected without waiting for 135 OCI calls
ds_target_connector_summary.sh --no-enrich

# Detailed view of "problematic" targets
ds_target_connector_summary.sh --no-enrich -D
```

### Alphabetical connector sorting in `ds_target_connector_summary.sh`

Connector groups are now sorted alphabetically by connector name in all output
formats (table, JSON, CSV). This makes the output consistent and easy to scan
regardless of the order in which OCI returns connectors.

---

## Changed

### `ds_target_connector_summary.sh` — connector grouping logic

- Connector OCID lookup now uses an OCID prefix filter
  (`startswith("ocid1.datasafeonpremconnector.")`) instead of membership in a
  pre-fetched connector list, making the result correct regardless of
  compartment scope.
- Fallback also checks `connection-option["on-prem-connector-id"]` and
  `connection-option["on-premise-connector-id"]` (both API field name variants).
- Label changed from `"No Connector (Cloud)"` to `"Cloud / Private Endpoint"`
  to more accurately describe cloud-native and Private Endpoint targets.

### `ds_target_connector_summary.sh` — removed options

`--enrich-connectors` and `--max-enrich-lookups` (previously undocumented
intermediate options) have been removed. The targeted post-grouping enrichment
replaces their purpose and requires no configuration.
