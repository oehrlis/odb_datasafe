# Release Notes v0.17.4

## Highlights

This patch release adds consistent target-scope flags to four `ds_target_*` scripts,
adds a new `--list` subcommand to `ds_target_audit_trail.sh` for audit trail status
overview, fixes a broken audit-trail start implementation, and closes several smaller bugs.

- Added `-A/--all` and `-r/--filter REGEX` scope flags to `ds_target_audit_trail.sh`,
  `ds_target_details.sh`, and `ds_target_move.sh`; added `-r/--filter REGEX` to
  `ds_target_delete.sh`.
- All four scripts now fall back to `DS_ROOT_COMP` when no scope flag is given, so
  `--filter REGEX` alone is a valid invocation (consistent with `ds_target_refresh.sh`).
- Added `--list`/`-l` subcommand to `ds_target_audit_trail.sh`: read-only per-target
  audit trail status view (trail lifecycle state + advisory note), with `--input-json`,
  `--save-json`, and `-f/--format table|json|csv` support.
- Fixed `ds_target_audit_trail.sh` audit-trail start: the script now lists trails per
  target then starts each by its audit-trail OCID (`--audit-trail-id`) using the correct
  OCI parameters; already-running trails (`COLLECTING`, `STARTING`, `RESUMING`) are skipped.
- Fixed `die` argument-order bug and suppressed-error-output issues in
  `ds_target_audit_trail.sh`.
- Fixed missing `shift` for `--stop-on-error` in `ds_target_delete.sh`.
- Added BATS test files for `ds_target_audit_trail.sh`, `ds_target_delete.sh`, and
  `ds_target_move.sh`; extended `ds_target_details.bats` with scope-flag tests.

## Added

### `ds_target_audit_trail.sh` — `--list` subcommand

A new read-only `--list`/`-l` subcommand provides a per-target audit trail
status overview without modifying any resource:

```text
TARGET                                       TRAIL STATE      NOTE
-------------------------------------------- ---------------- --------------------
prod_cdb01_CDBROOT                           COLLECTING       ok
prod_pdb01                                   STOPPED          needs restart
dev_cdb02_CDBROOT                            (no trail)       missing
```

**State → note mapping:**

| Trail state                | Note            |
|----------------------------|-----------------|
| `COLLECTING`               | ok              |
| `STARTING` / `RESUMING`    | starting        |
| `STOPPED`                  | needs restart   |
| `INACTIVE`                 | inactive        |
| `NEEDS_ATTENTION`/`FAILED` | needs attention |
| `DELETING` / `DELETED`     | deleted         |
| (no audit trail)           | missing         |

**New flags:**

| Flag                | Description                                              |
|---------------------|----------------------------------------------------------|
| `-l, --list`        | List audit trail states (read-only)                      |
| `--input-json FILE` | Read targets from `ds_target_list.sh --save-json` output |
| `--save-json FILE`  | Save selected target JSON payload                        |
| `-f, --format FMT`  | Output format: `table` (default), `json`, `csv`          |

`--input-json` avoids a second OCI target-database list call: only the
`audit-trail list` call (one per target) is needed. Compartment-id is
extracted from the cached target payload; a fallback `target-database get`
is used only for explicit-OCID targets that lack compartment-id in the cache.

`--input-json` and `--save-json` are also now accepted in **start mode**,
enabling a workflow where targets are pre-selected once and reused:

```bash
ds_target_list.sh --all --filter prod --save-json /tmp/prod_targets.json
ds_target_audit_trail.sh --list  --input-json /tmp/prod_targets.json
ds_target_audit_trail.sh --start --input-json /tmp/prod_targets.json
```

### Scope flags for `ds_target_*` scripts

All four scripts now accept the same scope flags established by `ds_target_refresh.sh`.
The shared helpers `ds_resolve_all_targets_scope`, `ds_validate_target_filter_regex`,
`ds_collect_targets_source`, and `ds_filter_targets_json` from `lib/oci_helpers.sh` are
used throughout.

| Script                     | `-A/--all` | `-r/--filter` |
|----------------------------|------------|---------------|
| `ds_target_audit_trail.sh` | ✓          | ✓             |
| `ds_target_details.sh`     | ✓          | ✓             |
| `ds_target_move.sh`        | ✓          | ✓             |
| `ds_target_delete.sh`      | —          | ✓             |

`-A/--all` selects all targets from `DS_ROOT_COMP` and is mutually exclusive with
`-c/--compartment` and `-T/--targets`. `-r/--filter REGEX` applies a substring regex
match against target display names. When no scope flag is given, all four scripts now
fall back to `DS_ROOT_COMP` (matching `ds_target_refresh.sh`), so `--filter REGEX`
alone is a valid invocation.

### `ds_target_details.sh` — unified target collection

`do_work()` now uses a single `ds_collect_targets_source` call for both the explicit-
target and compartment-scan branches, replacing two separate resolution paths. The dead
`list_targets_in_compartment()` function was removed.

### `ds_target_audit_trail.sh` — skip already-running trails

Audit trails already in `COLLECTING`, `STARTING`, or `RESUMING` state are now skipped
with an informational log message instead of causing a start-command failure. The
per-target and overall summary counters include a `skipped` field.

### BATS tests for new scope flags

Three new test files and one extended file covering the `--all`/`--filter` flags:

| File                                      | Tests                                                      |
|-------------------------------------------|------------------------------------------------------------|
| `tests/script_ds_target_audit_trail.bats` | help, mutual exclusion, regex validation                   |
| `tests/script_ds_target_delete.bats`      | help, regex validation, `--stop-on-error` shift regression |
| `tests/script_ds_target_move.bats`        | help, required `-D`, mutual exclusion, regex validation    |
| `tests/script_ds_target_details.bats`     | extended: scope flags, filter on input-json                |

## Fixed

### `ds_target_audit_trail.sh` — broken audit-trail start

The previous implementation called `oci data-safe audit-trail start
--target-database-id <target-ocid>`, which is rejected by the OCI CLI (exit 2:
"No such option: --target-database-id"). The correct flow is:

1. List audit trails for each target:
   `oci data-safe audit-trail list --compartment-id <compartment-ocid> --target-id <target-ocid> --all`
2. Start each trail by its own OCID:
   `oci data-safe audit-trail start --audit-trail-id <trail-ocid>`

Additional parameter fixes in the `audit-trail list` call:

- `--target-database-id` is not a valid option; the correct flag is `--target-id`.
- `--compartment-id` is required; the compartment OCID is now extracted from the
  target details fetched at the start of each loop iteration.
- The response is an `AuditTrailCollection` with shape `{"data": {"items": [...]}}`;
  the `jq` query is `(.data.items // .data)[]?.id` to handle both the collection
  and flat-list shapes.

Invalid CLI parameters (`--is-auto-queries-enabled`, `--update-last-archive-timestamp`,
`--audit-trail-type`, `--collection-frequency`) were also removed from the start call;
valid parameters for `audit-trail start` are `--audit-collection-start-time` and
`--is-auto-purge-enabled`.

### `ds_target_audit_trail.sh` — `"now"` not a valid RFC3339 datetime

`--audit-collection-start-time now` is rejected by the OCI CLI (exit 2: "Value now
is not in a supported datetime format"). When `START_TIME` is the default value
`"now"`, `start_audit_trails()` now converts it to a proper RFC3339 UTC timestamp
at runtime: `date -u +"%Y-%m-%dT%H:%M:%SZ"`. User-supplied values are passed
through unchanged.

### `ds_target_audit_trail.sh` — `die` argument order (two occurrences)

Two `die` calls had the numeric exit code as the first argument and the message as
the second, the opposite of the `die(message, [code])` signature:

- `die 1 "${failed_count} audit trail(s) failed to start"` → `die "..." 1`
- `die 1 "No targets available for audit trail start"` → `die "..." 1`

Both caused a "numeric argument required" shell error when the failure branch was
reached.

### `ds_target_audit_trail.sh` — suppressed OCI errors

`oci_exec … > /dev/null 2>&1` redirected both stdout and stderr to `/dev/null`,
silencing the `log_error` / `log_trace` output from `oci_exec` on OCI failures.
Changed to `> /dev/null` (stdout only) so error details are always visible.

### `ds_target_details.sh` — `--filter` not applied to `--input-json` results

`do_work()` called `ds_collect_targets_source` with a hard-coded `""` as the
4th (filter) argument in the `--input-json` branch, so `--filter REGEX` was
silently ignored when reading from a local JSON file. Fixed by passing
`"$TARGET_FILTER"` instead.

### `ds_target_delete.sh` — missing `shift` for `--stop-on-error`

The `--stop-on-error` case in `parse_args()` was missing `shift`, causing the
following positional argument to be silently consumed rather than parsed.
