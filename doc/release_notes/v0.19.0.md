# Release Notes v0.19.0

## Highlights

This minor release introduces two major features:

**Tag-based target scoping** — `--tag-filter` across all target operation
scripts enables OCI freeform and defined tag filtering. Tag data is already
present in every target response — no additional API calls are required.

**Enhanced service/port update** — `ds_target_update_service.sh` now supports
configurable service name templates, listener port updates, root target handling,
and optional derivation of service name and port directly from OCI PDB connection
strings.

- **New**: `--tag-filter EXPR` flag available in all `ds_target_*` scripts,
  `ds_tg_report.sh`, and `ds_target_connector_summary.sh`
- **New**: `ds_filter_targets_by_tags()` helper added to `lib/oci_helpers.sh`
- **New**: Multiple `--tag-filter` flags are combined with AND semantics
- **Enhanced**: `ds_target_update_service.sh` — template engine, port update, root handling, `--from-oci`
- **New**: `oci_lookup_pdb_connection()` helper in `lib/oci_helpers.sh`

## Added

### `--tag-filter` — tag-based target scoping

Filter targets by OCI freeform or defined tags before any operation (list,
refresh, delete, update, report, etc.).

#### Syntax

```bash
# Freeform tag — key=value
ds_target_list.sh --tag-filter "Environment=Production"

# Freeform tag — key presence only
ds_target_list.sh --tag-filter "Owner"

# Defined tag — namespace/key=value
ds_target_list.sh --tag-filter "DBSec/Classification=internal"

# Defined tag — key presence only
ds_target_list.sh --tag-filter "DBSec/ContainerType"

# Multiple filters (AND semantics) — repeat the flag
ds_target_list.sh --tag-filter "Environment=Production" \
                  --tag-filter "DBSec/Level=High"

# Combined with existing filters
ds_target_list.sh -r "cdb01" --tag-filter "Environment=Production"
```

#### Semantics

| Expression   | Matches                                            |
|--------------|----------------------------------------------------|
| `key=value`  | freeform tag `key` equals `value`                  |
| `key`        | freeform tag `key` is present (any value)          |
| `ns/key=val` | defined tag namespace `ns`, key `key` equals `val` |
| `ns/key`     | defined tag namespace `ns`, key `key` is present   |

Multiple `--tag-filter` expressions are AND-combined: a target must satisfy
all expressions to be included.

#### Library change — `ds_filter_targets_by_tags()`

New function in `lib/oci_helpers.sh`:

```bash
ds_filter_targets_by_tags TARGETS_JSON TAG_FILTER
```

- `TARGETS_JSON`: `{"data": [...]}` blob (standard OCI list response)
- `TAG_FILTER`: newline-separated expressions (one per `--tag-filter` flag)

Returns filtered `{"data": [...]}` with only targets satisfying every
expression. Empty or absent `TAG_FILTER` returns input unchanged.

The function is wired into `ds_collect_targets_source()` (new parameter 7)
and `ds_collect_targets()` (new parameter 5), so all scripts benefit from
consistent filter semantics without duplicating logic.

## Enhanced — `ds_target_update_service.sh`

### Template-based service naming

Service names are now derived from the target display name via configurable
brace-style templates rather than a hardcoded `_exa` derivation.

| Placeholder         | Expands to                         |
|---------------------|------------------------------------|
| `{pdb}` / `{PDB}`   | PDB name from display name (lower/upper) |
| `{cdb}` / `{CDB}`   | CDB name from display name (lower/upper) |
| `{cluster}` / `{CLUSTER}` | Cluster name (lower/upper)   |
| `{sid}` / `{SID}`   | Alias for cdb — useful in root templates |
| `{domain}`          | `DB_DOMAIN` value                  |

```bash
# Default template (backward-compatible result)
ds_target_update_service.sh -c prod --apply

# Custom template
ds_target_update_service.sh -c prod \
    --service-template "{PDB}_PAAS.{domain}" --apply

# PDB + root templates
ds_target_update_service.sh -c prod \
    --service-template "{pdb}_exa.{domain}" \
    --root-service-template "{cdb}.{domain}" --apply
```

Root targets (CDB$ROOT) require `--root-service-template` or are skipped
with a warning. Configure in `datasafe.conf`:

```bash
SERVICE_TEMPLATE="{pdb}_exa.{domain}"
ROOT_SERVICE_TEMPLATE="{cdb}.{domain}"
```

### Listener port update

```bash
# Explicit port (enables port update)
ds_target_update_service.sh -c prod --port 1565 --apply

# Use LISTENER_PORT from config
ds_target_update_service.sh -c prod --update-port --apply

# Port only (skip service update)
ds_target_update_service.sh -c prod --port 1522 --no-service-update --apply
```

### OCI PDB connection string lookup (`--from-oci`)

Automatically derives service name **and** port from the OCI PDB object's
`pdb-default` connection string (e.g. `host:1565/pdb01_PAAS.example.com`):

```bash
# Derive service+port from OCI PDB metadata
ds_target_update_service.sh -c prod --from-oci --apply

# PDB lookup in a separate compartment
ds_target_update_service.sh -c datasafe-comp \
    --pdb-compartment exacs-comp --from-oci --apply
```

Uses the new `oci_lookup_pdb_connection()` library function. Targets whose
PDB cannot be found in OCI are skipped with a warning.

### New library function — `oci_lookup_pdb_connection()`

```bash
oci_lookup_pdb_connection PDB_NAME COMPARTMENT_OCID
# Returns: "<service>|<port>" or exits 1 if not found
# Source:  lib/oci_helpers.sh (after ds_filter_targets_by_tags)
```

Queries `oci db pluggable-database list`, filters TERMINATED PDBs, and
parses the `pdb-default` connection string into service name and port.
