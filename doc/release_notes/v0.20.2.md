# Release Notes v0.20.2

## Highlights

Patch release fixing a long-standing issue where `systemctl stop` on a Data Safe
connector service left the CMAN process running. `cmctl shutdown` exits 0 without
actually terminating the process; the missing force-kill was only performed by
`oradba_dsctl.sh`'s `plugin_stop` logic. Both the service installer and the
bulk-uninstall script now integrate with `oradba_dsctl.sh` when a registry alias
can be resolved from `oradba_homes.conf`.

- **Fix**: `install_datasafe_service.sh` generates service files with
  `ExecStart/Stop/Reload` pointing to `oradba_dsctl.sh` when a registry alias is
  known. Service `Type` switches from `forking` to `oneshot + RemainAfterExit=yes`.
- **Fix**: `uninstall_all_datasafe_services.sh` gains a `stop_service` helper that
  prefers `oradba_dsctl.sh stop <alias>` and falls back to `systemctl stop` +
  `pkill` force-kill for clean bulk removal.
- **Change**: `ORADBA_BASE` default aligned to `${ORADBA_PREFIX:-/opt/oradba}` in
  both scripts, matching the `oradba_core.conf` convention.

---

## Fixed

### `bin/install_datasafe_service.sh` - ExecStop left CMAN process running

**Symptom.** After `systemctl stop oracle_datasafe_<connector>.service` the `cmgw`
process remained active. `systemctl status` showed the service as inactive but
`ps aux | grep cmgw` returned live processes.

**Root cause.** The generated service file used:

```ini
Type=forking
ExecStop=/path/to/oracle_cman_home/bin/cmctl shutdown -c <cman_name>
```

`cmctl shutdown` sends a shutdown signal and exits immediately with code 0, without
waiting for the process to terminate. The actual force-kill of remaining `cmgw`
processes is only performed by `oradba_dsctl.sh`'s `plugin_stop` function.

**Fix.** `generate_service_file` now resolves the registry alias for the connector
from `${ORADBA_BASE}/etc/oradba_homes.conf` via `lookup_registry_alias`. When an
alias is found the generated service file uses:

```ini
Type=oneshot
RemainAfterExit=yes
ExecStart=${ORADBA_BASE}/bin/oradba_dsctl.sh start <alias>
ExecStop=${ORADBA_BASE}/bin/oradba_dsctl.sh stop <alias>
ExecReload=${ORADBA_BASE}/bin/oradba_dsctl.sh restart <alias>
```

`Type=forking` was replaced with `Type=oneshot + RemainAfterExit=yes` because
`oradba_dsctl.sh` is not a daemon тАФ it starts/stops and returns. With `forking`
systemd would time out waiting for a child PID that never appears.

When no alias is found (e.g. `oradba_homes.conf` absent), the script falls back to
the previous `cmctl`-based service file with a WARNING message.

---

### `bin/uninstall_all_datasafe_services.sh` - stop left CMAN process running

**Symptom.** After `uninstall_all_datasafe_services.sh --uninstall` the `cmgw`
processes of removed connectors remained active, preventing a clean reinstall.

**Fix.** New `stop_service` function:

1. Checks whether `${ORADBA_BASE}/bin/oradba_dsctl.sh` is executable and an alias
   can be resolved for the connector.
2. If yes: calls `oradba_dsctl.sh stop <alias>` (handles force-kill internally).
3. If no: calls `systemctl stop` and then `pkill -f <cman_bin>` to eliminate
   remaining processes.

## Changed

### `ORADBA_BASE` default path

Both scripts previously used a customer-specific hardcoded path as the default for
`ORADBA_BASE`. The default is now:

```bash
ORADBA_BASE="${ORADBA_BASE:-${ORADBA_PREFIX:-/opt/oradba}}"
```

This matches the convention in `oradba_core.conf` (`ORADBA_PREFIX=/opt/oradba`).
Environments that set `ORADBA_PREFIX` or `ORADBA_BASE` via their Oracle profile
automatically pick up the correct path without any extra configuration.

### `CONNECTOR_BASE` in `uninstall_all_datasafe_services.sh`

`CONNECTOR_BASE` was previously computed as a local variable inside `list_services`.
It is now a global set at the top of the script so that `stop_service` and future
helpers share a single authoritative value.

## Migration: update existing service files

Existing service files generated by an older version still use `cmctl` directly.
To apply the fix:

```bash
# 1. Stop and remove existing services cleanly
sudo bin/uninstall_all_datasafe_services.sh --uninstall --force

# 2. Re-generate service files with oradba_dsctl.sh integration
bin/install_datasafe_service.sh --prepare -n <connector-name>

# 3. Install updated service files
sudo bin/install_datasafe_service.sh --install -n <connector-name>

# 4. Verify
sudo systemctl stop oracle_datasafe_<connector-name>.service
ps aux | grep cmgw  # must be empty
```

## Upgrade notes

No configuration changes required for environments that already set `ORADBA_BASE`
or `ORADBA_PREFIX` in their Oracle profile. Environments relying on the previous
default path must either set `ORADBA_BASE` explicitly or ensure `ORADBA_PREFIX` is
set before invoking the scripts.

Regenerate service files for any connector managed by these scripts to pick up the
`oradba_dsctl.sh` integration (see Migration section above).
