# Release Notes v0.20.3

## Highlights

Patch release fixing a silent failure in `install_datasafe_service.sh --install`
where a mismatch between the user specified in `--prepare` and the user passed to
`--install` was not detected. This caused an opaque systemd error `217/USER`
("Failed to determine user credentials") at service start instead of a clear
error at install time.

- **Fix**: `install_service()` validates `User=` in the prepared service file
  against `--user`; aborts with actionable instructions if they differ.
- **Fix**: Missing sudoers file now emits a warning instead of being silently skipped.
- **Fix**: `ExecStart` binary is checked for existence before installation.

---

## Root Cause

### Why `217/USER` appeared

`--install` was run with `--user oravw --group oravw`, but `--prepare` had
previously been run without `--user` (defaulting to `oracle`). The prepared
service file therefore contained `User=oracle`. `install_service()` copied it
unchanged.

systemd then tried to resolve `User=oracle` at service start. Because the
`oracle` OS user does not exist on this system (the Oracle software owner is
`oravw`), systemd reported:

```text
Failed to determine user credentials: No such process
Failed at step USER spawning /opt/oradba/bin/oradba_dsctl.sh: No such process
oracle_datasafe_exacc-wob-vwg-ha1.service: Failed with result 'exit-code'.
```

The "No such process" (ESRCH) is the errno returned by `getpwnam_r()` when the
NSS lookup finds no matching user entry.

### Why the sudoers file was not installed

`install_service()` looks for the sudoers file at
`$CONNECTOR_ETC/${OS_USER}-datasafe-${CONNECTOR_NAME}`. With `--user oravw` it
searched for `oravw-datasafe-exacc-wob-vwg-ha1`. The prepare had created
`oracle-datasafe-exacc-wob-vwg-ha1`. The file was not found and the installation
plan showed no sudoers entry — but no warning was emitted.

---

## Fixed

### `bin/install_datasafe_service.sh` - `User=` mismatch not detected

`install_service()` now reads `User=` from the prepared service file and compares
it to `$OS_USER` before copying anything:

```bash
file_user=$(grep -E '^User=' "${local_service}" | head -1 | cut -d= -f2- | tr -d ' ')
if [[ -n "${file_user}" ]] && [[ "${file_user}" != "${OS_USER}" ]]; then
    print_message ERROR "Service file specifies User=${file_user} but --user ${OS_USER} was given"
    print_message INFO "Re-prepare with correct user, then re-install:"
    ...
    return 1
fi
```

With this check, the mismatch is caught immediately with a clear error and the
exact commands needed to fix it, instead of surfacing as a cryptic systemd failure
after installation.

### `bin/install_datasafe_service.sh` - missing sudoers file silently skipped

When the expected sudoers file is absent and `--skip-sudo` was not set, a warning
is now printed:

```text
[WARNING]: Sudoers file not found: /path/to/oravw-datasafe-exacc-wob-vwg-ha1
[INFO]  Re-prepare to generate it: install_datasafe_service.sh --prepare -n ...
```

### `bin/install_datasafe_service.sh` - `ExecStart` binary not checked

The `ExecStart` binary path is now extracted from the prepared service file and
checked for existence before installation. If the binary is missing (e.g. because
`ORADBA_BASE` pointed to the wrong path during `--prepare`), a warning is emitted:

```text
[WARNING]: ExecStart binary not found or not executable: /opt/oradba/bin/oradba_dsctl.sh
[INFO]  If using oradba_dsctl.sh: set ORADBA_BASE and re-run --prepare
```

The installation plan now also shows `Service User` and `ExecStart` fields so
these values are visible before any changes are made.

## Correct workflow

```bash
# Step 1: prepare as oracle/oravw user (no root needed)
ORADBA_BASE=/appl/oracle/local/oradba \
  ./install_datasafe_service.sh --prepare -n exacc-wob-vwg-ha1 \
    --user oravw --group oravw

# Step 2: install as root
sudo ORADBA_BASE=/appl/oracle/local/oradba \
  ./install_datasafe_service.sh --install -n exacc-wob-vwg-ha1 \
    --user oravw --group oravw

# Step 3: verify
sudo systemctl status oracle_datasafe_exacc-wob-vwg-ha1.service
ps aux | grep cmgw
```

If `ORADBA_PREFIX` or `ORADBA_BASE` is exported in the user's Oracle profile,
the explicit `ORADBA_BASE=...` prefix is not needed.

## Upgrade notes

No configuration changes required. Re-run `--prepare` with the correct `--user`
and `--group` values whenever the OS user for the service differs from the default
(`oracle`). This ensures the service file, sudoers file, and README all carry the
correct username before installation.
