# Release Notes v0.20.1

## Highlights

Patch release fixing a regression triggered by the system upgrade of
`oci-cli` to 3.83.x on Python 3.14. Every `bin/ds_*.sh` script that
resolved a compartment by name (e.g. `ds_target_list.sh -C -v`) failed with
`Compartment not found` because the OCI CLI's `urllib3` dependency emitted
a `FutureWarning` on stderr that the OCI wrapper merged into stdout via
`2>&1`. The warning text ended up concatenated with the OCID returned to
the caller and broke a downstream `--query data[?name=='...']` lookup.

- **Fix**: `lib/oci_helpers.sh` now captures stdout and stderr
  independently in `oci_exec` / `oci_exec_ro` via a shared
  `_oci_run_capture` helper. Warnings emitted on stderr no longer reach
  the caller as data.
- **Defense-in-depth**: `lib/common.sh` exports `PYTHONWARNINGS=ignore`
  and `OCI_CLI_SUPPRESS_FILE_PERMISSIONS_WARNING=True` once at load time,
  so a forgotten `2>&1` elsewhere does not immediately break parsing.

All 17 `bin/ds_*.sh` scripts inherit the fix automatically via the shared
library.

## Fixed

### `lib/oci_helpers.sh` - stderr leakage into OCI command output

**Symptom.** `ds_target_list.sh -C -v` aborted with
`Compartment not found: <urllib3 warning text>...<OCID>` after upgrading
to `oci-cli` 3.83.x on Python 3.14.

**Root cause.** Both wrappers captured the OCI CLI with
`output=$("${cmd[@]}" 2>&1)` and echoed `$output` back to the caller.
That made every byte the OCI CLI wrote to stderr (Python deprecation
warnings, TLS notices, file-permission warnings) part of the data
stream consumed by downstream `--query` / `jq` operations.

**Fix.** New private helper `_oci_run_capture` runs the OCI CLI with
stdout captured into a variable and stderr captured into a temp file
(`mktemp` portable across BSD/GNU). On success it echoes stdout only;
stderr is surfaced at TRACE level for diagnostics. On failure both
streams are logged at TRACE and the original exit code is propagated.

`oci_exec` and `oci_exec_ro` now route through this helper. Behaviour
that callers rely on - dry-run handling in `oci_exec`, "always execute"
in `oci_exec_ro`, redacted-command logging, exit code propagation - is
preserved unchanged.

```text
# Before
output=$("${cmd[@]}" 2>&1)        # warnings + data interleaved
echo "$output"

# After (via _oci_run_capture)
stdout=$("$@" 2> "$stderr_file")  # streams separated
echo "$stdout"                    # data only
```

## Changed

### `lib/common.sh` - silence noisy Python warnings at library load time

Added two environment exports right after the multi-source guard:

- `PYTHONWARNINGS=ignore` - silences `FutureWarning`,
  `DeprecationWarning`, etc. emitted to stderr by the OCI CLI's bundled
  dependencies (urllib3, cryptography, ...).
- `OCI_CLI_SUPPRESS_FILE_PERMISSIONS_WARNING=True` - silences the
  informational notice when `~/.oci/config` has loose permissions.

Both respect a value the user has already exported (`${VAR:-default}`),
so a developer debugging a Python deprecation can still re-enable
warnings explicitly.

## Known follow-ups

Two call sites still merge stderr into stdout via `2>&1`. They are
already safe under the new `PYTHONWARNINGS=ignore` default because they
consume the captured output only for error-text pattern matching, not
JSON parsing - but routing them through `_oci_run_capture` for
consistency is recommended in a future release.

- `lib/oci_helpers.sh:135` - `check_oci_auth`
- `lib/oci_helpers.sh:1660` - `refresh_target`

## Upgrade notes

No configuration changes required. The fix is purely library-internal.
After pulling v0.20.1, simply re-run any previously failing
`ds_*.sh` invocation.

If you have local scripts that source `lib/common.sh` and rely on
Python warnings being visible (rare), override the defaults explicitly:

```bash
export PYTHONWARNINGS=default
source lib/common.sh
```
