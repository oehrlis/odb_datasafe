# Release Notes v0.18.3

## Highlights

This patch release fixes two issues related to targets in DELETED state:
DELETED/DELETING targets are no longer flagged as medium-severity health
issues, and re-registering a target whose previous entry is in DELETED state
is now allowed.

- **Fixed**: `ds_target_list.sh` health mode reported DELETED and DELETING
  targets as `TARGET_UNEXPECTED_STATE` / `MEDIUM`. They are now reported as
  `TARGET_DELETING` / `LOW` with "No action required".
- **Fixed**: `ds_target_register.sh` blocked registration when a DELETED target
  with the same display name existed. Registration now proceeds for DELETED
  name matches.

---

## Fixed

### DELETED/DELETING targets flagged as MEDIUM health issues

`ds_target_list.sh --mode health` (and `--mode problems`) evaluates the
lifecycle state of every target. States outside `HEALTH_NORMAL_STATES`
(`ACTIVE,UPDATING` by default) fell through to the `TARGET_UNEXPECTED_STATE`
catch-all with `MEDIUM` severity, including DELETED and DELETING targets.

DELETED and DELETING targets require no operator action — OCI removes them
automatically. Flagging them as `MEDIUM` added noise to health reports and
inflated medium-issue counts in the report mode.

**Fixed**: added an explicit `DELETED | DELETING` case before the wildcard,
emitting issue code `TARGET_DELETING` at `LOW` severity:

```text
ISSUE             SEVERITY  CLUSTER  SID     TARGET               STATE    DETAILS
TARGET_DELETING   LOW       exa01    cdb01a  exa01_cdb01a_CDBROOT DELETED  Target deletion in progress or complete; will be removed by OCI
                                                                            No action required
```

### Re-registration blocked by DELETED target with the same name

`check_target_exists()` in `ds_target_register.sh` searched for any target
matching `display-name` regardless of lifecycle state. If a DELETED target
shared the display name of a new target being registered, the registration
was rejected:

```text
[WARN]  Target already exists. Use a different display name or remove existing target first.
```

DELETED targets are essentially gone from an operational standpoint. OCI
retains the record for a period before purging it, but a new target with the
same name can be registered without conflict.

**Fixed**: `check_target_exists()` now filters out DELETED targets from the
match:

```bash
# Before
select(."display-name" == "$DISPLAY_NAME")

# After
select(."display-name" == "$DISPLAY_NAME" and ."lifecycle-state" != "DELETED")
```

Registration proceeds normally when the only name match is a DELETED target.
