# Release Notes v0.18.3

## Highlights

This patch release fixes three issues related to targets in DELETED/DELETING
state polluting health analysis and blocking registration, and adds Database
Vault detection to `ds_database_prereqs.sh`.

- **Fixed**: `ds_target_list.sh` health mode reported DELETED and DELETING
  targets as `TARGET_UNEXPECTED_STATE` / `MEDIUM`. They are now reported as
  `TARGET_DELETING` / `LOW` with "No action required".
- **Fixed**: `ds_target_list.sh` counted DELETED/DELETING targets in the
  per-SID root/PDB tallies, causing spurious `SID_DUPLICATE_ROOT` / `HIGH`
  issues for SIDs that had one active root and one deleted root.
- **Fixed**: `ds_target_register.sh` blocked registration when a DELETED target
  with the same display name existed. Registration now proceeds for DELETED
  name matches.
- **Added**: `ds_database_prereqs.sh` detects Database Vault (DV) before
  running any DDL. If DV is configured and enabled, the scope is skipped with
  a warning and manual instructions; other scopes continue normally.

---

## Fixed

### DELETED/DELETING targets flagged as MEDIUM health issues

`ds_target_list.sh --mode health` (and `--mode problems`) evaluates the
lifecycle state of every target. States outside `HEALTH_NORMAL_STATES`
(`ACTIVE,UPDATING` by default) fell through to the `TARGET_UNEXPECTED_STATE`
catch-all with `MEDIUM` severity, including DELETED and DELETING targets.

DELETED and DELETING targets require no operator action — OCI removes them
automatically. Flagging them as `MEDIUM` added noise to health reports and
inflated medium-issue counts in the report mode.

**Fixed**: added an explicit `DELETED | DELETING` case before the wildcard,
emitting issue code `TARGET_DELETING` at `LOW` severity:

```text
ISSUE             SEVERITY  CLUSTER  SID     TARGET               STATE    DETAILS
TARGET_DELETING   LOW       exa01    cdb01a  exa01_cdb01a_CDBROOT DELETED  Target deletion in progress or complete; will be removed by OCI
                                                                            No action required
```

### Spurious SID_DUPLICATE_ROOT caused by DELETED targets

The per-SID root/PDB counts that drive `SID_DUPLICATE_ROOT` and
`SID_MISSING_ROOT` included all targets regardless of lifecycle state. A SID
with one active CDBROOT and one DELETED CDBROOT was counted as having two
roots, producing a spurious `HIGH` issue:

```text
SID has duplicate CDB roots   HIGH   55   55   13.6%   Review duplicate root registrations...
```

Once OCI purged the DELETED records the issue disappeared, confirming the
root cause.

**Fixed**: DELETED and DELETING targets are now excluded from the root/PDB
tallies. Only active registrations are counted when checking structural
completeness per SID.

### Re-registration blocked by DELETED target with the same name

`check_target_exists()` in `ds_target_register.sh` searched for any target
matching `display-name` regardless of lifecycle state. If a DELETED target
shared the display name of a new target being registered, the registration
was rejected:

```text
[WARN]  Target already exists. Use a different display name or remove existing target first.
```

DELETED targets are essentially gone from an operational standpoint. OCI
retains the record for a period before purging it, but a new target with the
same name can be registered without conflict.

**Fixed**: `check_target_exists()` now filters out DELETED targets from the
match:

```bash
# Before
select(."display-name" == "$DISPLAY_NAME")

# After
select(."display-name" == "$DISPLAY_NAME" and ."lifecycle-state" != "DELETED")
```

Registration proceeds normally when the only name match is a DELETED target.

## Added

### Database Vault detection in `ds_database_prereqs.sh`

When Database Vault (DV) is configured and enabled, a SYSDBA connection cannot
create users or grant system privileges — the operations silently fail or raise
errors, with no clear indication that DV is the cause.

`ds_database_prereqs.sh` now queries `DBA_DV_STATUS` before running any DDL
for each scope (ROOT or individual PDB):

```sql
SELECT COUNT(*) FROM DBA_DV_STATUS
WHERE NAME IN ('DV_CONFIGURE_STATUS', 'DV_ENABLE_STATUS')
  AND STATUS = 'TRUE';
```

A count of 2 (both configured and enabled) triggers the warning and skips that
scope. If DV is not installed, `WHENEVER SQLERROR CONTINUE` silently absorbs
the `ORA-00942` and the script proceeds normally.

When DV is detected, the output is:

```text
[WARN]  Database Vault is configured and enabled on PDB=MYPDB.
[WARN]  SYSDBA cannot create users or grant system privileges when Database Vault is active.
[WARN]  Manual steps are required for this scope:
[WARN]    1. Connect as DV_ACCMGR (or account manager role) to create user: DS_ADMIN
[WARN]    2. Connect as DV_OWNER or SYS to grant the required Data Safe privileges
[WARN]  Skipping automated prerequisites for scope: PDB=MYPDB
```

Other scopes in the same `--all` run are unaffected and continue normally.

Note: a full DV-aware implementation (passing DV_ACCMGR credentials) was
considered but deferred — DV_ACCMGR alone is insufficient for all required
grants; DV_OWNER or SYS is also needed. Clear guidance via the warning message
is the preferred approach.

## Testing

```bash
make test

# Verify DELETED/DELETING targets appear as LOW in health mode, not MEDIUM
ds_target_list.sh --mode health

# Verify no spurious SID_DUPLICATE_ROOT when deleted roots exist
ds_target_list.sh --mode health --issue "SID has duplicate CDB roots"

# Verify re-registration of a deleted target succeeds (dry-run)
ds_target_register.sh --compartment <comp> --sid <sid> --pdb <pdb> \
    --connector <conn> --dry-run

# Verify DV detection warning (on a DV-enabled database)
ds_database_prereqs.sh --pdb MYPDB -P "<secret>"
# Expected: WARN messages and scope skipped, exit 0
```
