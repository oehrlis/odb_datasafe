# Release Notes v0.20.0

## Highlights

Minor feature release introducing `ds_target_reregister.sh` for seamless PDB
relocation handling in Oracle Data Safe. When a pluggable database is moved to a
new ExaCC cluster, a different host, or renamed, the existing Data Safe target can
now be updated in-place — no delete-and-recreate required. Credentials can be
refreshed in the same operation.

Also fixes a silent failure in `ds_target_activate.sh` where a single failing
target aborted the entire bulk activation run.

- **New**: `ds_target_reregister.sh` — update cluster, host, SID, PDB, service, and
  credentials after a PDB relocation
- **Fix**: `ds_target_activate.sh` — ERR trap no longer aborts bulk activation on
  first failing target

## Added

### `ds_target_reregister.sh` — Re-register a target after PDB relocation

New script to update the connection metadata of an existing Oracle Data Safe
target when the underlying PDB has been relocated. Replaces the manual
delete-and-recreate workflow.

**What it updates (selectively):**

| Field          | OCI attribute                         | Flag                             |
|----------------|---------------------------------------|----------------------------------|
| VM Cluster     | `databaseDetails.vmClusterId`         | `--cluster`                      |
| PDB OCID       | `databaseDetails.pluggableDatabaseId` | `--pdb` or `--cluster`           |
| SID (CDB name) | display name component                | `--sid`                          |
| PDB name       | display name component                | `--pdb`                          |
| Service name   | `databaseDetails.serviceName`         | `--service` or `--from-oci`      |
| Listener port  | `databaseDetails.listenerPort`        | `--port` or `--from-oci`         |
| Display name   | `displayName`                         | auto-derived or `--display-name` |
| Description    | `description`                         | `--description`                  |
| Credentials    | `credentials`                         | `--ds-secret`                    |

**Key design decisions:**

- **PUT semantics**: current `databaseDetails` is loaded from OCI and patched;
  unchanged fields (e.g. `databaseType`, `infrastructureType`) are preserved.
- **Display name schema**: auto-derived as `<cluster>_<cdb>_<pdb>` (or
  `<cdb>_<pdb>` when no cluster), consistent with `ds_target_register.sh`.
  The unchanged parts are taken from the current target name.
- **Credentials**: updated only when `--ds-secret` (or `--secret-file`) is
  provided. Root targets (`_CDBROOT` suffix) automatically receive the
  common-user prefix (default: `C##`). Credential update is a second OCI call
  after the structural update.
- **`--from-oci`**: queries `oci db pluggable-database list` for the PDB's
  `pdb-default` connection string and extracts service name and port automatically.
- **Dry-run by default**: changes are shown but not applied unless `--apply` is given.

**Typical usage:**

```bash
# PDB moved to new cluster - preview
ds_target_reregister.sh --target exa01_cdb01_MYAPP --cluster exa02

# PDB moved to new cluster - apply with service from OCI + credential refresh
ds_target_reregister.sh --target exa01_cdb01_MYAPP \
    --cluster exa02 --from-oci --ds-secret <secret> --apply

# Only PDB renamed on same cluster
ds_target_reregister.sh --target exa01_cdb01_OLDNAME --pdb NEWNAME --apply
```

**Reusable library functions (no new lib additions required):**

- `oci_resolve_vmcluster_by_name()` — cluster OCID resolution
- `oci_lookup_pdb_connection()` — PDB service/port from OCI (`--from-oci`)
- `ds_collect_targets()` — target name-to-OCID resolution
- `oci_exec()` / `oci_exec_ro()` — OCI CLI wrapping with dry-run support
- `ds_write_cred_json_file()` — credential JSON for OCI update

## Fixed

### `ds_target_activate.sh` — ERR trap aborted bulk activation on first failure

The `do_work()` loop called `activate_single_target` without an error guard.
When a target failed (e.g. connectivity issue), the ERR trap from
`setup_error_handling()` fired immediately, aborting all remaining targets.

The function already tracked failures in `FAILED_COUNT` and returned `1` — the
loop simply had to suppress the trap propagation:

```bash
# Before (aborts on first failure)
activate_single_target "$target_ocid" "$current" "$total"

# After (consistent with ds_target_refresh.sh)
if ! activate_single_target "$target_ocid" "$current" "$total"; then
    :
fi
```

All targets are now processed regardless of individual failures. The script
exits with code 10 at the end if `FAILED_COUNT > 0`, so the caller can detect
partial failures.
