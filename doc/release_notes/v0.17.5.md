# Release Notes v0.17.5

## Highlights

This patch release standardises the wait-state option across all nine mutating
scripts, adds VM cluster name resolution via OCI structured search, improves
debug observability for OCI CLI configuration and loaded config files, and fixes
several shell-correctness and diagnostic issues in target registration.

- Unified `--wait-state STATE` option on all mutating scripts. Default is now
  **async** (return immediately after submitting the OCI API call); pass
  `--wait-state SUCCEEDED` (or any valid work-request state) to block.
- New `oci_resolve_vmcluster_by_name()` library helper resolves a VM cluster
  display name to both its OCID and `compartment-id` in a single structured-
  search round-trip, eliminating a previous two-step lookup.
- `check_oci_auth()` now logs the active OCI config file/profile, loaded
  `datasafe.conf` paths, and `DS_ROOT_COMP` at DEBUG level.
- Fixed shell-quoting of OCI commands in trace/error logs (`printf '%q '`).
- Fixed subshell variable propagation bug in `ds_target_register.sh` cluster
  resolution; connector compartment fallback now emits explicit warnings.

## Added

### Unified `--wait-state STATE` across all mutating scripts

All nine mutating scripts now expose a single, consistent option:

```text
--wait-state STATE    Wait until the OCI work request reaches STATE.
                      Default: empty (return immediately after submit).
```

| Script                            | Previous behaviour             | v0.17.5                         |
|-----------------------------------|--------------------------------|---------------------------------|
| `ds_target_register.sh`           | blocking poll to `ACTIVE`      | async; `--wait-state ACTIVE`    |
| `ds_target_refresh.sh`            | `--wait` / `--no-wait` bool    | `--wait-state STATE`            |
| `ds_target_update_connector.sh`   | `WAIT_FOR_COMPLETION` bool     | `--wait-state STATE`            |
| `ds_target_activate.sh`           | `--wait-for-state STATE`       | `--wait-state STATE`            |
| `ds_target_update_credentials.sh` | `--wait-for-state STATE`       | `--wait-state STATE`            |
| `ds_target_update_service.sh`     | `--wait-for-state STATE`       | `--wait-state STATE`            |
| `ds_target_update_tags.sh`        | `--wait-for-state STATE`       | `--wait-state STATE`            |
| `ds_target_delete.sh`             | hardcoded `SUCCEEDED` blocking | async; `--wait-state SUCCEEDED` |
| `ds_target_move.sh`               | no wait support                | `--wait-state STATE`            |

`WAIT_STATE` is normalised to uppercase at parse time. The OCI CLI passthrough
remains `--wait-for-state "$WAIT_STATE"` (OCI CLI naming convention).

`lib/oci_helpers.sh`: `ds_refresh_target()` updated to branch on `WAIT_STATE`:
non-empty → blocking with state label in log; empty → async.

**Examples:**

```bash
# Fast async (default — all scripts)
ds_target_refresh.sh --all

# Block until work request is SUCCEEDED
ds_target_refresh.sh --all --wait-state SUCCEEDED

# Activate and get earliest confirmation
ds_target_activate.sh --all --apply --wait-state ACCEPTED

# Delete with explicit blocking (previous default behaviour)
ds_target_delete.sh -T mydb01 --force --wait-state SUCCEEDED
```

### `oci_resolve_vmcluster_by_name()` — single-call cluster resolution

New function in `lib/oci_helpers.sh`. Resolves a VM cluster display name to
both its OCID and `compartment-id` via a single `search resource
structured-search` call (tries `VmCluster` first, then `CloudVmCluster`).
Returns a JSON object `{"id":"ocid1...","compartment-id":"ocid1..."}`.

`bin/ds_target_register.sh` `validate_inputs()` now calls this function
directly in the parent shell, extracting both values in one round-trip and
falling back to `resolve_vm_cluster_ocid()` (DB-list scan) only when the
structured search yields no result.

### Debug observability improvements

- `lib/common.sh`: added `_DATASAFE_CONF_FILES` global; `load_config()`
  appends each successfully loaded config file path so that a summary can
  be emitted at startup.
- `lib/oci_helpers.sh`: `check_oci_auth()` now logs at `DEBUG` level:
  - Active OCI CLI config file, profile, and region.
  - All loaded `datasafe.conf` file paths (from `_DATASAFE_CONF_FILES`).
  - Resolved `DS_ROOT_COMP` value (or `(not set)`).

## Fixed

### `lib/oci_helpers.sh` — plaintext bundle password exposed in logs (security fix)

`ds_generate_connector_bundle()` passed `--password "$password"` as an inline
OCI CLI argument. `oci_exec()` then logged the full command at TRACE level **and**
at ERROR level (shown on any failure), exposing the plaintext bundle encryption
password in log files and console output.

**Fix:** Added `_oci_redact_cmd()` — a one-pass sanitizer that replaces the value
following `--password` in the command array with `****` before formatting the string
for log output. Applied to all six log sites across `oci_exec()`, `oci_exec_ro()`,
and `ds_refresh_target()`:

| Log site | Level | Before | After |
| -------- | ----- | ------ | ----- |
| `oci_exec` — trace | TRACE | `… --password mysecret …` | `… --password **** …` |
| `oci_exec` — dry-run | INFO | `… --password mysecret …` | `… --password **** …` |
| `oci_exec` — error | ERROR | `… --password mysecret …` | `… --password **** …` |
| `oci_exec_ro` — trace | TRACE | same | masked |
| `oci_exec_ro` — error | ERROR | same | masked |
| `ds_refresh_target` — trace/error | TRACE/ERROR | same | masked |

The actual execution array passed to the shell is unchanged. Extend the `case`
block in `_oci_redact_cmd()` to cover additional sensitive flags if needed.

### Shell-quoting of OCI commands in trace/error logs

`oci_exec()` and `oci_exec_ro()` in `lib/oci_helpers.sh` previously used
`${cmd[*]}` to log the OCI command, which joined array elements with spaces
and stripped quoting from multi-word arguments (e.g. `--query-text "query
VmCluster …"`). The logged command could not be copy-pasted and re-executed.
Fixed by using `$(printf '%q ' "${cmd[@]}")` so each element is shell-escaped.
Same fix applied to the trace log in `ds_refresh_target()`.

### `ds_target_register.sh` — subshell variable propagation bug

`validate_inputs()` called `oci_resolve_vmcluster_by_name()` inside a
`$(...)` command substitution, which ran in a subshell. The assignments to
`CLUSTER_OCID` and `derived_compartment` were invisible to the parent shell,
leaving `derived_compartment` always empty. The script then fell through to
the host-derivation path unnecessarily. Fixed by calling the function directly
and assigning from its JSON output via `jq`.

### `ds_target_register.sh` — ExaCC PDB OCID not required

For ExaCC targets (where `CLUSTER_OCID` is set), a failed `pluggableDatabaseId`
lookup is now a warning rather than a fatal error. The Data Safe API accepts
registration using `vmClusterId` + `serviceName` + `listenerPort` without a
PDB OCID, matching legacy behaviour in `register_datasafe_pdb_v1.0.0.sh`.
Non-ExaCC registrations still require and enforce a resolved PDB OCID.

### `ds_target_register.sh` — connector compartment fallback warning

When `DS_ROOT_COMP` and `DS_CONNECTOR_COMP` are both unset, the connector
lookup previously fell back silently to the target database compartment
(usually wrong). The fallback now emits two `[WARN]` lines directing the user
to configure the missing variable. The `die` message on connector-not-found
includes the searched compartment OCID for easier diagnosis.
