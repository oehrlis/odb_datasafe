# Release Notes - v0.22.0

**Release Date:** 2026-06-28
**Type:** Minor (M2 - Security Hardening)

## Summary

M2 eliminates credential exposure on argv and disk, tightens Oracle privilege grants
to the CIS/STIG minimum, reduces IAM and installer attack surface, and adds runtime
guards for Python version and vendor file integrity.

## Changes

### Credential Handling

- `bin/ds_target_register.sh`: `-P/--ds-secret` now emits a deprecation warning
  (SEC-002). The password is still accepted for one-release migration but users are
  directed to `--secret-file` instead, which avoids exposure in `ps` and shell history.
- `lib/oci_helpers.sh` `ds_generate_connector_bundle()`: bundle unlock password is no
  longer passed on the OCI CLI argv. It is written to a `mktemp`/`umask 077` temp
  file and passed as `file://` — the temp file is removed via a `RETURN` trap (SEC-003).
- `bin/ds_target_register.sh` `register_target()`: registration JSON payload (which
  contains the DB credentials) is now written to a `mktemp`/`umask 077` file with an
  automatic `RETURN` trap for cleanup. On failure, a `jq`-redacted copy is preserved
  for debugging and the real payload is deleted immediately (SEC-004).

### Config Sourcing Trust

- `lib/common.sh` `load_config()`: config files are now validated before sourcing.
  World/group-writable files and files owned by an untrusted user (neither `root` nor
  the invoking user) are rejected with a warning and skipped (SEC-005).
  Successfully sourced files are logged at `INFO` level.

### Installer Hardening

- `bin/install_datasafe_service.sh`: removed `2>/dev/null || true` from both `chown`
  calls — ownership failures now surface as errors (SEC-006).
- `bin/install_datasafe_service.sh`: sudoers `journalctl` rule changed from trailing
  wildcard (`-u $SERVICE_NAME*`) to exact `--unit=$SERVICE_NAME` form; both
  `/bin/journalctl` and `/usr/bin/journalctl` paths included for distro compatibility
  (SEC-007).
- `bin/uninstall_all_datasafe_services.sh`: `pkill -f` and `pgrep -f` patterns now
  anchored with `^` to prevent matching unrelated processes (SEC-008).

### Oracle Privilege Reduction

- `sql/create_ds_admin_prerequisites.sql`: `PASSWORD_LOCK_TIME` corrected from
  `299/86400` (~5 min) to `1` (1 day, CIS minimum). `INACTIVE_ACCOUNT_TIME` changed
  from `DEFAULT` to `35` days (CIS maximum). `CONTAINER=ALL` added to `CREATE PROFILE`
  so common users in PDBs receive the same profile (ORA-003, ORA-004).
- `sql/create_ds_admin_user.sql`: `GRANT CONNECT, RESOURCE` replaced with
  `GRANT CREATE SESSION` in both grant paths (normal and password-reuse exception).
  `RESOURCE` role historically grants `UNLIMITED TABLESPACE` and object-creation
  system privileges that Data Safe does not require (ORA-005).
- `sql/extension_comprehensive.sql`: removed `HOST echo ... > /tmp/oradba_logdir_*.sql`
  round-trip that wrote and executed a predictable world-readable temp file. Replaced
  with a static `DEFINE LOGDIR = '.'`; callers can override via SQL*Plus
  `-DEFINE LOGDIR=/path` (ORA-009).

### Dependency Guards

- `bin/ds_connector_create.sh`: added Python 3.8+ version check and non-empty file
  guard before executing vendor `setup.py` (DEP-004, DEP-012).
- `bin/ds_connector_update.sh`: same guards added before `setup.py` update invocation.

### Documentation

- `doc/oci-iam-policies.md`: added security recommendations and `# WARNING` comments
  on tenancy-wide `manage` statements; operators are directed to the compartment-
  scoped `where target.compartment.id=` form already shown in the JSON examples
  (ORA-013, ORA-014).
- `doc/database_prereqs.md`: added "Privilege Surface by Grant Mode" section documenting
  each `--grant-mode` value, the features it enables, and the system privileges granted.
  Operators are recommended to use the minimum mode for their deployment (D-3).

## Upgrade Notes

- **journalctl sudoers rule has changed.** The `NOPASSWD` rule in `/etc/sudoers.d/`
  now uses `--unit=` instead of `-u *`. Re-run `--install` (or manually update the
  sudoers file) to apply the new, narrower rule.
- **GRANT RESOURCE removed.** Existing DS admin accounts created by previous versions
  retain the `RESOURCE` role. To reduce privileges, revoke it manually:
  `REVOKE RESOURCE FROM <ds_admin_user>;`. The updated SQL template applies only on
  fresh provisioning.
- **PASSWORD_LOCK_TIME now 1 day.** Existing profiles must be updated via
  `ALTER PROFILE DS_USER_PROFILE LIMIT PASSWORD_LOCK_TIME 1;` to pick up the
  hardened value.
