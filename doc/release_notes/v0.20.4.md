# Release Notes v0.20.4

## Highlights

Patch release fixing three runtime failures encountered when installing Data Safe
connector services in environments where `ORACLE_BASE` is not exported (e.g. running
as root) or where the connector was installed under a non-standard path.

- **Fix**: `ORACLE_BASE` not auto-detected as root; connector base path is now
  discovered from a list of known candidate directories.
- **Fix**: User= mismatch between `--prepare` and `--install` now triggers automatic
  re-generation of service files instead of aborting with an error.
- **Fix**: `oradba_dsctl.sh` fails if `/var/log/oracle/` does not exist; the generated
  service file now sets `ORADBA_LOG` to `${CONNECTOR_HOME}/log`, and that directory
  is created during `--install`. `oradba_dsctl.sh` itself gains a graceful fallback
  to `/tmp` when the log directory cannot be created.

---

## Root Causes

### Issue 1: ORACLE_BASE not set when running as root

`CONNECTOR_BASE` defaults to `${ORACLE_BASE:-/u01/app/oracle}/product`. On systems
where the Oracle software owner is not `oracle` (e.g. `oravw`) or when running as
`root` (which does not inherit user-profile exports), `ORACLE_BASE` is empty and
the default resolves to `/u01/app/oracle/product`. If the connector lives under
`/appl/oracle/product/` the path check fails and `validate_connector()` errors out.

### Issue 2: User= mismatch causing re-prepare requirement

`--prepare` (run as a regular user) and `--install` (run as root) may carry different
`--user` arguments if the operator re-runs them independently. The v0.20.3 check
correctly detected the mismatch but required a manual re-prepare step. Running
`prepare_service()` inline (as root) is safe and avoids the round-trip.

### Issue 3: Missing /var/log/oracle/ crashes oradba_dsctl.sh

`oradba_dsctl.sh` defines its log file as:

```bash
LOGFILE="${ORADBA_LOG:-/var/log/oracle}/${SCRIPT_NAME%.sh}.log"
```

Under `set -euo pipefail`, any `echo >> "${ORADBA_LOG_FILE}"` call made by
`oradba_common.sh` fails immediately with `No such file or directory` when
`/var/log/oracle/` does not exist. The service unit therefore fails at `ExecStart`.

The fix has two layers: the service file sets `ORADBA_LOG=${CONNECTOR_HOME}/log`
(a path that always exists inside the connector's own tree), and `oradba_dsctl.sh`
gains a safety net that falls back to `/tmp` when `mkdir -p` on the configured
directory fails.

## Fixed

### `bin/install_datasafe_service.sh` - connector base auto-discovery

New function `find_connector_base()` probes a list of candidate base directories
when the connector is not found under the default `CONNECTOR_BASE`:

```bash
candidates=(
    "${ORACLE_BASE:+${ORACLE_BASE}/product}"
    "/appl/oracle/product"
    "/u01/app/oracle/product"
    "/u01/oracle/product"
    "/opt/oracle/product"
)
```

In `main()`, if `CONNECTOR_BASE` resolves to the default value and the connector
directory is not found there, `find_connector_base()` is called and `CONNECTOR_BASE`
is updated to the first matching path before `validate_connector()` runs.

### `bin/install_datasafe_service.sh` - auto-regenerate on User= mismatch

`install_service()` previously aborted when `User=` in the prepared service file
did not match `--user`. It now calls `prepare_service()` inline (re-generating
the service file with the correct user) and `chown`s the result:

```bash
prepare_service
chown "${OS_USER}:${OS_GROUP}" "${CONNECTOR_ETC}"/* 2>/dev/null || true
```

The installation proceeds immediately after without requiring a separate
`--prepare` run.

### `bin/install_datasafe_service.sh` - ORADBA_LOG in generated service file

When `REGISTRY_ALIAS` is set (i.e. `oradba_dsctl.sh` is used), `generate_service_file()`
now emits:

```ini
Environment="ORADBA_LOG=${CONNECTOR_HOME}/log"
```

This overrides the default `/var/log/oracle` with a path that is local to the
connector installation and does not require system-wide directory creation.

`install_service()` creates `${CONNECTOR_HOME}/log` with the correct ownership
before copying the service file to `/etc/systemd/system/`.

### `src/bin/oradba_dsctl.sh` (oradba repo) - graceful log directory fallback

After computing `LOGFILE`, the script now ensures the directory exists before
the `set -euo pipefail` strict mode can trigger on a write:

```bash
if [[ ! -d "${LOGFILE%/*}" ]]; then
    mkdir -p "${LOGFILE%/*}" 2>/dev/null || LOGFILE="/tmp/${SCRIPT_NAME%.sh}.log"
fi
```

If the directory cannot be created (e.g. running without write permission to
`/var/log/`), the log file silently falls back to `/tmp/oradba_dsctl.log`.

## Correct workflow (updated)

```bash
# Step 1: prepare as the correct service user
ORADBA_BASE=/appl/oracle/local/oradba \
  ./install_datasafe_service.sh --prepare -n exacc-wob-vwg-ha1 \
    --user oravw --group oravw

# Step 2: install as root (User= mismatch → auto-regenerate; log dir → auto-created)
sudo ORADBA_BASE=/appl/oracle/local/oradba \
  ./install_datasafe_service.sh --install -n exacc-wob-vwg-ha1 \
    --user oravw --group oravw

# Step 3: verify
sudo systemctl status oracle_datasafe_exacc-wob-vwg-ha1.service
ps aux | grep cmgw
```

With `ORACLE_BASE` unset (root session): the connector under `/appl/oracle/product/`
is now discovered automatically without requiring `--base`.

## Upgrade notes

Re-run `--install` for any connector where the service was previously failing with
the `/var/log/oracle` error. The new `ORADBA_LOG` variable in the service file and
the auto-created log directory resolve the issue without further configuration.

No other configuration changes are required.
